Every DLQ incident asks the same four questions.

What died?

Which payloads, from which exchange, rejected how many times — and why?

Can I put it back?

Replay to the original queue — but only the messages that are safe to retry.

What about the rest?

Park the poison messages somewhere they can't hurt anyone.

Who touched what?

The incident review is coming. You need a paper trail, not a memory.

The Management UI shows you queue counts. Beyond that your options are rabbitmqadmin get — which consumes the message while you look at it — or a one-off script written at 3 a.m. QueueLens is the tool you wish you'd installed before the incident.

the contract

One rule: a failed action must never lose a message.

Every design decision in QueueLens follows from it. These aren't features — they're guarantees, each with an enforcement point in code.

§ 01

Browsing never consumes

Non-destructive preview with requeue — read 220 messages, all 220 stay exactly where they were.

§ 02

Publish before ack

Replay removes the original only after the broker confirms the publish. Unroutable publishes bounce back as errors, never silence.

§ 03

Dry-run is mandatory

Bulk actions first count exactly what will be touched; execute runs on that exact set via a one-shot confirmation token.

§ 04

Deletes are deliberate

Type-to-confirm, Admin role only. There is no "oops" path.

§ 05

Everything is on the record

An attempt event is written before every action and an outcome after. If the attempt can't be persisted, the action is refused.

§ 06

Ambiguity fails closed

A message fingerprint matching zero or multiple messages aborts the action instead of guessing.

the second rule

An observability tool must never become the incident.

QueueLens is deliberately lazy toward your broker — it reads metadata, not messages, unless you ask.

  • ✓Dashboard reads metadata only — queue lists and counts from the Management API; message bodies are never fetched in the background.
  • ✓Preview is manual and bounded — bodies are read only when you open a queue, capped, and requeued immediately.
  • ✓No automatic payload scanning — no crawler walks your queues; filters run only inside a user-initiated, size-capped dry-run.
  • ✓Topology is cached — the exchanges/bindings snapshot is served from a 30-second cache.
  • ✓Diagnostics on demand — the connection test fires when you click it, not on a timer.
  • ✓Alerts check counts, not contents — queue statistics on a configurable interval; message bodies are never touched.
  • ✓Auto-refresh is optional — the dashboard's 30-second refresh can be switched off; every other screen loads on navigation only.

Govern

quickstart

Sixty seconds to your first rescued message.

demo — bundled broker 
# one command, demo DLQs included
$ git clone https://github.com/talaatmagdyx/queuelens
$ cd queuelens
$ docker compose up --build

# → console at http://localhost:8000/app
# → admin / change-me

Ships with a RabbitMQ broker and pre-seeded dead-letter queues to practice on.

production — your cluster 
$ docker run --rm -p 8000:8000 \
    -e QUEUELENS_RABBITMQ_URL='amqps://…' \
    -e QUEUELENS_ADMIN_PASSWORD='…' \
    -e QUEUELENS_SECRET_KEY='…' \
    ghcr.io/talaatmagdyx/queuelens:latest

Multi-stage image · non-root · precompiled UI · healthchecked. Every variable documented in CONFIGURATION.md.
